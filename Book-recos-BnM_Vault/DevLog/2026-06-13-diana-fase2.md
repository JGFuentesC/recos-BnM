## Qué se hizo

### Fase 1 — Conexión API real en Library.jsx
- Reemplazados los mocks con `GET ${VITE_API_URL}/api/collections?userId={uid}` usando `useAuth()` y `currentUser.getIdToken()` para el Bearer token
- Spinner de carga mientras el fetch corre; pantalla de error "Error al cargar tu biblioteca" si falla
- `MOCK_ITEMS` se mantienen como fallback solo si `VITE_API_URL` no está definida (`USE_API = !!import.meta.env.VITE_API_URL`)
- Cleanup con flag `cancelled` para evitar `setState` en componente desmontado

### Enriquecimiento asíncrono de ítems sin título
- El endpoint `GET /api/collections` devuelve ítems sin `title`, `cover` ni `genres` (solo campos de Firestore via `toPublic`)
- Normalización defensiva: `genres: Array.isArray(item.genres) ? item.genres : []` evita el crash de `CollectionItem` en `item.genres.map()`
- Ítems sin `title` entran en el Set `enriching`; se dispara `GET /api/content/{contentId}` en paralelo para cada uno
- Mientras enriquecen: **skeleton card** animado (shimmer) en lugar de `CollectionItem`; al llegar la respuesta se actualiza el ítem y desaparece el skeleton

### Fix 401 en PATCH/DELETE — fetch interceptor
- `CollectionItem.jsx` hace sus propias llamadas `fetch` sin Bearer token (no puede modificarse — ownership Edgar/JC/Monserrat/Marina según CLAUDE.md)
- Solución: interceptor en `window.fetch` instalado mientras `Library` está montada; inyecta `Authorization: Bearer {token}` solo en PATCH y DELETE a `/api/collections/`
- Limpieza en el cleanup del `useEffect` — restaura el `window.fetch` original al desmontar

### Toast "✓ Nota guardada"
- Toast en esquina inferior derecha (fondo verde oscuro `#14532d`) tras PATCH exitoso
- Variante roja (`toastErr`) para errores
- Animación `fadeInUp` 0.25s inyectada en `<head>` junto con `spin` y `shimmer`

### Bug fix — toast aparecía en el segundo blur, no en el primero
- **Causa raíz doble:**
  1. `Library.jsx` nunca sincronizaba `personalNote` de vuelta a `items` tras un PATCH — el guard de `CollectionItem` (`note === item.personalNote`) siempre fallaba, disparando un PATCH por cada blur subsiguiente
  2. `clearTimeout(showTimer)` cancelaba el toast del primer PATCH cuando llegaba el segundo dentro de los 2 segundos del delay original
- **Fix:**
  - Eliminado el delay de 2 segundos — toast aparece inmediatamente al recibir `response.ok`
  - El interceptor parsea `init.body` para extraer `personalNote` y llama `setItems` actualizando el ítem en el estado de Library — así `item.personalNote` coincide con `note` en el siguiente blur y el guard bloquea el PATCH duplicado
  - `showTimer` eliminado; solo queda `hideTimer` para ocultar el toast tras 3 segundos
